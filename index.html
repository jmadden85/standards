IIN Front End Standards

Creating and mantaining coding standards is not only important for keeping up with best practices, it helps immensely with future maintanence, onboarding, and allowing others to quickly understand your code.

While browsers may not care what your code looks like (so long as it's valid), other developers will. The way code looks is a big part of how we (developers) understand it. Think of how often you've opened up code someone else wrote, and changed the indentation to match what you like, or gone through it and added white space/comments to help you understand and quickly be able to make changes throughout the code. When everyone is writing code that looks different, each person needs to take more time to visually parse the code before understanding it. When everyone's code looks the same, everyone can grasp it and work with it faster and easier.

If you think of code as an actual language (human language, not computer language), you realize that you want to be able to speak this language in a way that others familiar with the language can easily understand. When you write me an email, I should be able to read through it briefly and quickly grasp the content. Furthermore, if I forward that email to someone else, they too should be able to grasp it quickly.
