IIN Front End Standards

Creating and mantaining coding standards is not only important for keeping up with best practices, it can be immensely helpful with future maintanence, onboarding, and helping other devs to quickly understand your code.

While browsers may not care what your code looks like (so long as it's valid), other developers do. The way code looks is a big part of how we understand it. How many times have you opened up code someone else wrote, and changed the indentation to match what you like, or gone through it and added white space/comments to help you understand, and be able to swiftly make changes throughout the code. When everyone is writing code that looks different, each person needs to take more time to visually parse the code. When code is clean and consistent, everyone can grasp, and work with it faster and easier.

If you think of code as an actual language (english, japanese, french etc.), you realize that you want to be able to speak/write this language in a way that others familiar with the language can easily understand. When you write me an email, I should be able to read through it briefly and quickly ascertain the point. Furthermore, if I forward that email to someone else, they should be able to do the same. Our code needs to work the same way.

Indentation

For all code languages, we require indentation to be done via soft tabs (using the space character). Hitting Tab in your text editor needs to be equivalent to four spaces.

Naming Conventions

It's preferred to use names that are relative to what the object/element is rather than what it looks like. For example, an class of 'greenDropDown' doesn't make much sense when the color gets changed to blue and someone decides we don't want it to be a dropdown anymore. However if this dropdown was for the nav, use something like 'navMenu'. For all class names, ids, and function names you should use camelCase. i.e. the-frog should be theFrog, or this_id should be thisId, or stop-showing-this-stuff-to-me should be stopShowingThisStuffToMe.

Express concerns on performance issues early

As a front end developer it's your job to communicate any performance issues you see which may arise from different interactive features or pretty visuals. Explain what the issues are in as much detail as you can and, provide alternative solutions. Doing this as early as possible prevents us from losign time when we discover this issue after we're done building.

Cross Browser Compatibility

At IIN we support mobile, chrome, safari, opera, firefox, and IE 8+. Please check your code in all of these browsers before passing it to QA. While we don't explicitly support IE 7, make sure your code functions reasonably in IE 7.

Bootstrap

At IIN we use bootstrap. Bootstrap is a handy front end framework which provides us with a nice way to keep a common theme across our sites. If you are unfamiliar with bootstrap, there is plenty of great documentation which can be found here and here.

HTML Standards & Style Guide

HTML is the main component of any webpage, it's also the last thing a browser has to fallback on when CSS/Javascript aren't working properly. As such our HTML needs to be valid, clean, and structured in such a way that the page's information can still be passed on sans CSS/Javascript.

General Markup Guidelines

First and foremost, it is important that developers always strive to use markup which is symantically accurate to the content being displayed. These are general rules and best practices for structuring markup.

HTML5 has provided us awesome new elements. They all help solve different issues like lack of semantics due to 'divitis', and slowly but surely killing flash. When possible/logical/semantical we should always try and use these new elements.

Overview

  • Tables shouldn't be used for page layout.
  • Use double quotes around all attributes
  • When creating attributes which are specifically used for functionality (Javascript, or backend tasks) use the data- format
    1. <span data-postedBy="bob">This document is too large</span>
    3. function hidePostsIDontLike(name) {
    4. var postsToHide = $("[data-postedBy=" + name + "]");
    5. postsToHide.hide(0);
    6. };
    7. hidePostsIDontLike("bob");
  • Inline styles are to be avoided, use a CSS file.
  • Inline functionality is to be avoided, use a JS file.
  • All form fields should have a label with a for attribute that matches the respective form field's id. This is helpful for UX, mobile, and accessibility.
  • Use the meta viewport tag, very important to mobile friendliness.
    1. <meta name="viewport" content="width=device-width">
  • With large chunks of HTML inside of an element it can be helpful to use a comment on the closing tag to indicate what element you're closing.
  • Javascript files should be put at the bottom of the page. Browsers load content from the top down and javascript does not allow parallel downloads (nothing else can be downloaded while a javascript file is being downloaded).
  • Always use the appropriate case for elements. Do not use all caps or lowercase, instead use the text-transform CSS property.
  • Lists are lists, so they should be in a UL, OL, or DL.
  • When creating tables (sorry) make use of the THEAD, TBODY, and TH tags when appropriate.

CSS Standards & Style Guide

CSS, enter the presentation layer. CSS, not only does it make our sites look great, it's also daunting/confusing enough to back end developers to prevent them from doing our job.

As CSS is the presentation layer, your HTML should be coded in such a manner that it's not dependant on CSS. The CSS is meant to make the information you're trying to share with your HTML pretty. All of your content needs to be comprehensible even with CSS disabled.

Shorthand

CSS shorthand is preferred thanks to it's terseness, and ability to make faster edits to already existing values such as padding and margin.

Responsive Design & Mobile First

First of all, if you don't know or aren't super comfortable with responsive design, check this out. It's a quick read, and it really covers all the bases on responsive design.

If you do know and are comfortable with responsive design, make sure to always be thinking mobile first. If you start on a new project, and nobody is talking mobile, bring it up, make sure it's a priority. If it's not going to be, talk to the PM, the designer, or your manager and figure out why.

That being said, say goodbye to pixels. For width including margin and padding, you should use percentages. As of now, and until the 'px vs em' debate has settled we'll be using em's for font-size.

Images

For any background gradients it's preferred to use CSS3, however if a background image is needed use something larger than 1x1 pixels.

Use of sprites is heavily encouraged, cut's back on http requests as well as making hover/focus/active states easy.

Typography

As previously mentioned we use bootstrap, which should handle our typography needs.

Cross browser

CSS3 is still being developed, as such not all browsers were sure how to handle it. All the good browsers wanted to get their hands on it early to give their users a better experience and help flush out CSS3. In order to do this they created vendor prefixes to add these features that might not yet be part of a form specification.

In order to support different browsers the vendor prefixes need to be used when implementing CSS3 features.

General CSS Guidelines

There are a couple big things that can be done to really help reduce CSS clutter. The first one is to try and cascade these cascading styles as much as you can. Cascading not only helps mantain code, it plays a big part in mantaining a common look and feel across the site. If you are given a design for a page in which all the font sizes are different from the rest of the site, the nav has different background colors, and things are generally not alike to the rest of the site, push back on the design. Talk to the PM or the designer, figure out why/if this page is supposed to be different from the other pages.

The second important thing is to try and keep specificty low. If you aren't familiar with the complex inner workings of specificity check this out, it's a pretty good explanation.

When possible try and use a single container class followed by plain html selector. Even if you can't do this, try and avoid ID's and never use !important. This keeps the specificity low, and when you do get that crazy looking page and the PM tells you 'This page is supposed to look crazy, we're agile we can make crazy pages whenever we want.', you won't need to prefix all your selectors #crazyPage #crazyPageWrapper #crazyPageContent in order to override the styles.

Overview

  • Keep specificity low, try and use class selectors and avoid ID's. Never use !important.
  • Put your CSS in sections and comment when a new section begins with what the section is styling. Keeps things organized, and easy to figure out.
  • Stay mobile friendly, widths in percentages, font sizes in ems.
  • Use vendor prefix for cross browser support.
  • Use sprites when background images are needed.
  • Use CSS3.
  • Use shorthand CSS.
  • Write speed friendly selectors. Avoid the * wildcard selector and don't qualify ID selectors (e.g.div#someId) or class selectors (e.g. div.someClass).
  • Familiar with the box model? Get more familiar.

Javascript Standards & Style Guide

Don't call me Java bro.

Javascript is used to add functionality and different interactions to/with your content. In the same way that your HTML needs to not be CSS dependant, it also needs to not rely on your javascript to display the content.

General Javascript Guidelines

All of your js should be housed in exteranl javascript files, which should be placed right before the closing body tag for performance.

As with HTML, always try to class/method/function/variable names sensible. If you have a modal window for sign in, call it "signInModal" oppose to "theModal".

DRY or Don't Repeat Yourself

Always try and make functions which can be reused and are generalized. A few good ways to do this are to use parameters, closures, and return values. So if you're working on something which you think may be reused later, like a gallery, instead of hard coding it's width, photos, methods of switching photos write a function which takes parameters for all of these things. This can greatly reduce code overhead.

Global Variables & Module Pattern

Global variables are easily one of Javascript's worst features. Not only do they pollute the namespace, they are also accessible by everything under the sun, and since they are the last thing on the scope chain they are bad for performance. Global variables are to be avoided at all costs. It doesn't really make sense to create a variable until it's going to be used, so all variables should be declared at the top of the function/method they are going to be used in.

The module pattern is a great way to avoid global variables. The basic idea is to create a single object global variable which will contain all of your method's.

For more information on the module pattern read this.

Javascript Equality Operators & Coercion

Javascript comes loaded with 2 sets of equality operators. One of them works the way you'd expect an equality operator to work, and is the only one you should ever use.

                            1 === 1 //true
                            1 === '1' //false
                            1 !== '1' //true
                            1 !== 1 //false

The other operator is evil, breaks code, and doesn't work like you'd expect.

                            1 == 1 //true
                            1 == '1' //true
                            1 != '1' //false
                            1 != 1 //false
                            ' \t\r\n ' == 0 //true

The blame for this lies within javascript's coercion "feature". The == operator attempts to coerce the values, so mixing types (String, Number, Boolean, Array, Object, Null, Undefined) will unexpectedly return very incorrect results. The rules for coercion are far too stupid to bother explaining/understanding. It's basically like saying a banana and a potato are the same thing because they are both types of food you've eaten.

Coercion can alse be problematic when it comes to concatenation.

                            1 + 'cat' //"1cat"
                            1 + 3 + 'cat' //"4cat"
                            1 + '3' + 'cat' //"13cat"
                            '1' + 3 + 'cat' //"13cat"
                            'cat' + 10 + 30 //"cat1030"

Semicolon Insertion

Javascript attempts to fix broken code by automatically inserting semicolons, where it thinks they should go. This can be problematic and should be avoided. Semicolons are to explicitly be put at the end of functions, method calls, breaks, and return statements.

                            return
                            {
                                name : 'joe'
                            }

The above code looks as if it should return an object with a name member. However due to an automatic placement of a semicolon on the return, the function drops out there and nothing is returned. As such no bugs are thrown and this bug is a mystery. This can be avoided by putting the opening { on the same line as the return.

                            return {
                                name : 'joe'
                            }

For loops, if statements, switch, while loops

All of your block conditionals and loops should be done in this format.

                            for ( var i = 0, t = that.length; i < t; i++ ) {

                                var thisEl = that[t];

                                if ( thisEl.id == 'elImLookingFor' ) {

                                    //Get elements height, split the string, removing px
                                    //Parse the string to an integer, add 100 use coercion to conver back to string with px on the end and set the height
                                    thisEl.style.height = parseInt(thisEl.style.height.split('px')[0]) + 100 + 'px';

                                }

                            }

Please use the extra spaces in between the parens which contain the conditions. For and While loops should have their iterators ( var i = 0 ) declared as local variables oppose to global variables ( i = 0 ).

An if statement which only has a single line inside it doesn't need to use curly brackets. However, it's not very costly to add them and, it can help prevent missing things like this.

                            if ( that.length )
                                doThis();
                                andThat();

This looks like.

                            if ( that.length ) {
                                doThis();
                                andThat();
                            }

It actually means this.

                            if ( that.length ) {
                                doThis();
                            }
                            andThat();

This can be a difficult to spot.

Be certain to never let your switch cases fall through by forgetting to add breaks.

                            switch (expression) {

                               case label1:
                                  statements1
                                  break;

                               case label2:
                                  statements2
                                  break;

                               case labelN:
                                  statementsN
                                  break;

                               default:
                                  statements_def
                                  break;

                            }
